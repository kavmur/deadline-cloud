## Scripts

### API Change Detection

This directory contains scripts for API change detection using [griffe](https://mkdocstrings.github.io/griffe). This is used by our github CI actions so we can notify code authors of any unintentional API changes before they are merged. If any breaking changes are necessary, code authors should follow our [conventional commits](../CONTRIBUTING.md#conventional-commits) policy to mark the change as such.

#### Overview

The API change detection system uses a two-step approach:

1. We use the Griffe CLI to detect semantic breaking changes (parameter changes, removals, etc.)
2. We use API snapshots (generated by Griffe CLI) to detect any API changes missed by Griffe, which includes: additions, removals, and modifications

#### Scripts

##### `generate_api_snapshot.py`

Generates a complete API snapshot of the deadline package using griffe serialization. Supports generating snapshots from git references for dynamic baseline comparison.

```bash
# Generate snapshot from current working directory using hatch command (recommended)
hatch run docs:generate-api-snapshot /tmp/snapshot.json

# Generate snapshot from a git reference (e.g., mainline branch)
hatch run docs:generate-api-snapshot /tmp/baseline_snapshot.json mainline

# Or run directly with custom path
hatch run docs:python scripts/generate_api_snapshot.py /tmp/snapshot.json
```

##### `validate_api_snapshot.py`

Validates the current API against a baseline snapshot to detect any changes.

```bash
# Validate against a baseline snapshot using hatch command (recommended)
hatch run docs:validate-api-snapshot /tmp/baseline_snapshot.json

# Or run directly with custom path
hatch run docs:python scripts/validate_api_snapshot.py /tmp/baseline_snapshot.json
```

#### Usage Workflow

##### When Making API Changes

When making any API changes, follow these guidelines:

**For new APIs (additions):**
- Use conventional commit syntax to mark new features:
  ```bash
  git commit -s -m "feat: Add new job submission API endpoint"
  ```

**For breaking changes:**
- Mark your commit as a breaking change following [conventional commit syntax](../CONTRIBUTING.md#conventional-commits):
  ```bash
  git commit -s -m "feat!: Remove deprecated parameter from create job API"
  # or
  git commit -s -m "feat: Remove deprecated parameter

  BREAKING CHANGE: The 'legacy_mode' parameter has been removed from create_job()"
  ```

**For non-breaking modifications:**
- Use appropriate conventional commit types:
  ```bash
  git commit -s -m "fix: Correct return type annotation for get_job_status"
  git commit -s -m "docs: Update docstring for submit_job function"
  ```

##### Testing Locally

Before pushing, you can test locally:

```bash
# Check for breaking changes against mainline branch using hatch command (recommended)
hatch run docs:check-api-breaks --against origin/mainline

# Or check for breaking changes against the latest tag using hatch command
hatch run docs:check-api-breaks

# Or run griffe directly
hatch run docs:griffe check -s src deadline --against origin/mainline

# Generate and validate API changes against mainline
hatch run docs:generate-api-snapshot /tmp/baseline.json mainline
hatch run docs:validate-api-snapshot /tmp/baseline.json
```

#### What Gets Detected

##### Breaking Changes (griffe check)
- ❌ Removing public APIs/functions/classes
- ❌ Adding required parameters to existing functions
- ❌ Changing parameter types or defaults
- ❌ Removing parameters
- ❌ Moving positional parameters
- ❌ Changing return types (when supported)

##### All Changes (snapshot validation)
- 🆕 **ADDED**: New APIs, functions, classes, methods, attributes
- 🗑️ **REMOVED**: Deleted APIs, functions, classes, methods, attributes  
- 🔄 **MODIFIED**: Changed function signatures, parameters, return types, decorators, class bases, attribute types/values

### Running Tests with Docker for File and Directory Permissions

We have some unit tests that require being run in a specific docker container that is set up for testing with different users. Those unit tests are marked with `docker`, and the `run_sudo_tests.sh` script is provided to facilitate this testing. The script builds the Docker image using the Dockerfile located in `testing_containers/localuser_sudo_environment/`, and then runs the container.

#### Usage
Execute the script from the root of the repository.
```
./scripts/run_sudo_tests.sh --build
```

Or, you can run the hatch script:
```
hatch run test_docker
```

Please make sure that you have the necessary permissions to execute the script.
